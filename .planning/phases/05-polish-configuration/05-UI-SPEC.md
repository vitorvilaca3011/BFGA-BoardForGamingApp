## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #0A0A0A | Window base, board surround, negative space behind floating panels |
| Secondary (30%) | #111111 | Settings panel, roster strip, bottom bar, elevated chrome |
| Accent (10%) | User-selected shared presence color | Selected laser swatch fill, current color preview chip, roster avatar fill, remote cursor identity, remote laser identity |
| Destructive | #C42B1C | Disconnect or close-destructive affordances only |

Accent reserved for: selected presence-color swatch, current-color preview, roster avatar fill, cursor tint, and laser head/trail/ping tint for that same user identity. Never use the presence color for every toolbar icon, panel border, generic hover state, or unrelated drawing controls.

Existing neutral shell tokens stay unchanged: `BgSurface #0D0D0D`, `BgOverlay #161616`, `TextPrimary #FAFAFA`, `TextSecondary #B0B0B0`, `BorderDefault #2A2A2A`, `AccentWhite #FFFFFF`.

## Visual Hierarchy And Interaction States

Primary focal point: live laser head and trail on canvas remain the first visual read during pointing. Phase 5 changes color ownership only; it must not change Phase 3 size, zoom, or fade behavior.

Secondary focal point: settings panel `LASER COLOR` section. Show a visible section label, a compact current-color preview, and a 4x4 swatch grid reusing existing `color-swatch` sizing and hover motion.

Tertiary focal point: roster avatars and other presence surfaces echo the same chosen color so the user reads one identity across roster, cursor, and laser.

Interaction states:
- Color configuration lives in the existing settings panel only. No toolbar quick picker, no duplicate source of truth.
- Picker shape: reuse the existing 24x24 swatch button pattern from `PropertyPanel.axaml`; selected swatch gets a 2px `AccentWhite` ring plus unchanged fill color.
- Choosing a swatch updates the persisted shared presence color immediately. The same color must propagate to roster, remote cursor, and laser presence without waiting for restart.
- Laser color remains independent from `SelectedStrokeColor`; changing drawing color must not change the stored presence color.
- Switching away from Laser Pointer, pointer exit, or pointer capture loss must send the same deactivation behavior as release: stop emission immediately and let remaining local trail finish fading naturally.
- Remote silence for about 3 seconds behaves like a synthetic release: stop new points immediately, keep the rendered trail fading naturally.
- Peer disconnect removes that peer's remote laser state immediately with no ghost trail left behind.
- Accessibility fallback: swatch buttons sit under a visible `LASER COLOR` label and helper copy. Do not replace this with unlabeled toolbar-only icon affordance.

Scale contract:
- Swatch size: 24px square, matching existing property-panel swatches.
- Selected swatch ring: 2px light border, no extra glow.
- Current-color preview chip: 24px square minimum.
- Settings panel width and placement remain aligned with current panel shell: 260px width, right-side floating panel rhythm preserved.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Choose Laser Color |
| Empty state heading | Laser color ready |
| Empty state body | Open Settings and choose a color for your multiplayer presence. The same color will appear in roster, cursor, and laser feedback. |
| Error state | Couldn't apply laser color. Choose the swatch again; if collaborators still see the old color, reconnect to refresh presence. |
| Destructive confirmation | none: no destructive action in this phase |

Helper-text rules:
- Section label stays `LASER COLOR` to match existing uppercase settings language.
- Helper copy should say `Used for roster, cursor, and laser` or equivalent explicit scope line.
- Do not use generic labels like `Save`, `Apply`, or `OK` for instant-update swatches.
